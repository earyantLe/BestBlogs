### 🟩 技术 / 架构（TECH · ARCH）

---

**Prompt Caching 的工程正确使用方法** — METHOD ⭑×4
① 缓存的是 KV 张量而非原始文本，完全不损失模型智能；② 前缀须达 1024 token 才能激活；③ 推理模型（o1/o3）必须用 Responses API（思维链 token 不跨 API 持久化）；④ Cache Key 本质是「分片键」，控制请求路由局部性，优化后可将命中率从 60% 提升至 87%；⑤ 上下文窗口管理中，大步长截断比频繁微调更省钱。
- 个人价值：在 API 开发中正确配置 Caching 可节省大量成本，这套知识很少被集中讲解。
- 可行动性：HIGH ｜ 出处：[Build Hour：Prompt Caching](https://www.youtube.com/watch?v=tECAkJAI_Vk)（94 分）

---

**上下文工程（Context Engineering）是比提示词工程更根本的学科** — PRINCIPLE
Agent 的输出质量不仅取决于模型能力，更取决于「在每一步推理中放了什么信息、过滤了什么信息」。四类失效模式：爆发（超出窗口）、冲突（指令矛盾）、投毒（错误信息被纳入上下文）、噪声（不相关信息降低信噪比）。任务间独立时用裁剪（快但丢失信息），任务相互依赖时用摘要（慢但保留连续性）。
- 项目关联：BestBlogs 的 Daily Brief 生成链路中，上下文管理直接影响输出质量，值得系统优化。
- 可行动性：HIGH ｜ 出处：[Build Hour：代理记忆模式](https://www.youtube.com/watch?v=WsGVXiWzTpI)（93 分）

---

**WebSocket + Responses API：Agent 循环提速 40%** — METHOD
WebSocket 持久连接 + `previous_response_id` 缓存机制让 Codex Agent 循环端到端提速 40%，核心是「只处理增量，不重算历史」。当推理速度提升 15 倍（65→1000 TPS），API 服务层本身成为 Agent 循环的新瓶颈——需要 WebSocket 解除这层约束。
- 个人价值：构建 Agent 时的关键架构决策，直接影响用户体验和 API 成本。
- 可行动性：HIGH ｜ 出处：[通过 Responses API 中的 WebSockets 加速智能体工作流](https://openai.com/index/speeding-up-agentic-workflows-with-websockets)（93 分）

**多 Agent 编排：分流架构 + 编排逻辑与计算容器解耦** — PRINCIPLE ⭑×2
① 用一个分类器 Agent 将任务路由到专注的子 Agent，每个子 Agent 只看到与其任务相关的工具和上下文，避免通用 Agent 的工具混乱。② 编排逻辑（决定做什么）必须与计算容器（执行做什么）解耦，是构建生产可用 Agent 的最关键架构决策。Skills API + Git 版本控制是解决多 Agent 并行部署中 Prompt 一致性的工程级答案。
- 项目关联：BestBlogs 的多步内容处理流水线（分析→评分→翻译→早报）正是多 Agent 编排的典型场景。
- 可行动性：HIGH ｜ 出处：[Build Hour: AgentKit](https://www.youtube.com/watch?v=sAitLFLbgDA)（93 分）、[OpenAI Agents SDK](https://www.youtube.com/watch?v=tK32trvj_b4)（92 分）

**语音 AI 三种设计模式分类法** — METHOD
① Voice-to-Action（用户说话触发任务执行）；② Systems-to-Voice（系统主动用语音推送信息）；③ Voice-to-Voice（跨语言实时翻译对话）。语音转语音（S2S）架构优势在于单模型端到端理解音频，能捕捉笑声、叹息等非语言信号，三模块串联（ASR + LLM + TTS）无法实现。Preamble 技巧：让模型边思考边说话，把沉默等待变成对话节奏，是语音 Agent 的「人性化桥接」方案。
- 可行动性：MEDIUM ｜ 出处：[通过 API 新模型推进语音智能](https://openai.com/index/advancing-voice-intelligence-with-new-models-in-the-api)（94 分）

**Evals 范式：从「考试模式」到「工作模式」** — PRINCIPLE ⭑×2
GDP Eval 宣告评估范式转移：学术基准满分不等于能胜任真实工作。GPT-5 在真实工作任务胜率从不足 20% 升至接近 40%，18 个月翻倍。多 Agent 系统应优先评估单个节点（node-level eval），再评估端到端链路，而不是只做整体成功/失败判断。Harness 的选择本身就是能力测量的一部分，错误的脚手架会系统性低估模型真实能力。
- 项目关联：BestBlogs 内容质量评分体系可以借鉴「工作模式 Eval」，用真实用户行为而非人工标注来评估内容质量。
- 可行动性：HIGH ｜ 出处：[Evals 实践](https://www.youtube.com/watch?v=YEaKXjHENyQ)（93 分）

**部署模拟：用真实对话前缀预测模型行为** — PRINCIPLE
用真实生产对话前缀重新生成候选模型回复，把安全评估覆盖度从「人工 prompt 设计能力」解耦。传统合成 eval 数据集被模型识别为「正在被评估」的概率接近 100%（评估感知失真），意味着传统 eval 可能严重低估真实不良行为率。「计算器劫持」这类边缘行为只有真实上下文才能触发，标准 eval 完全无法发现。

## 二、双轨汇总

### 🎯 对项目（BestBlogs）

**内容处理链路 → Codex 异步委托**
BestBlogs 的大批量内容处理（AI 评分、翻译、标签、早报生成）是「睡前下令」工作流的理想场景。可以用 Symphony 规范管理多个 Codex 会话，用 `agents.md`/`CLAUDE.md` 约定降低上下文对齐摩擦，用 Agent-native 遥测提升可观测性。关联卡：睡前下令 [77]、agents.md [78]、Symphony [98-100]。

**图像生成升级 → Images 2.0 思考模式**
BestBlogs 的封面图、早报配图生成目前是单轮提示，Images 2.0 的思考模式（先搜索规划再生成）可以显著提升图文一致性和排版质量。关联卡：[48]、[49]、[103]。

**Daily Brief / 早报产品 → Workspace Agents 方向参考**
ChatGPT Workspace Agents 的「零工程门槛构建 Agent」方向与 BestBlogs 早报 skill 的运营目标高度重合，可以参考其「触发器 + 过程 + 权限 + 治理边界」四要素框架审查现有 skills 的完整性。关联卡：[12]、[40]。

**自我改进机制 → 内容质量反馈闭环**
「从业者纠正 → 结构化发现 → Eval 目标 → Codex 工程任务 → PR」这个 Playbook 可以用于 BestBlogs 内容评分 Agent 的持续改进。目前缺少结构化的「发现 → 改进」闭环。关联卡：[84]、[85]。

**多 Agent 编排 → 分流架构应用**
BestBlogs 内容处理流水线（分析→评分→翻译→早报）可以引入「分流 Agent + 专用子 Agent」架构，将不同类型内容路由到专门的处理 Agent，提升效率和质量。关联卡：[56]、[82]。

### 🌱 对个人（认知 / 思维 / 视野 / 技能）

**Codex 使用心法体系**
「假设失败是你的错」→「agents.md 约定文件」→「睡前下令工作流」→「Symphony 目标驱动」构成一套从认知到工程的完整 Codex 使用方法论。核心是：不断优化你给 AI 的「上下文质量」，而非期待 AI 更聪明。

**上下文工程作为核心能力**
上下文工程（Context Engineering）正在成为 AI 时代最重要的工程技能——理解四类失效模式，掌握裁剪 vs 摘要决策，设计 Prompt Caching 分片键，这些技能的价值正在超越传统的算法能力。

**「授权过度」的警醒**
AI 时代最重要的元认知：保留批判性思考，把 AI 当「放大器」而非「替代者」。「势头悖论」提醒我们：专注力比工具能力更稀缺，主动选择「不做什么」比「多用工具」更重要。

**跨域连接的思维方式**
Erdős 突破的启示：在某个领域陷入瓶颈时，引入其他领域的工具和视角可能直接突破。AI 本身就是跨域连接器，善用 AI 的人会比他人更频繁地经历这种「跨域突破」。
