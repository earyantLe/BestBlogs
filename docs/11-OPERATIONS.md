# 11-OPERATIONS.md

更新时间：2026-07-08

> 本文是公开同步版运维原则，只保留架构约束、灰度、监控、告警、回滚和安全边界。真实环境变量、内部地址、告警路由、供应商账号、on-call 细节、事故记录和逐步执行脚本应放在私有 runbook 中。

## 原则

BestBlogs 的运维设计遵循：

1. **默认可回滚**：新能力优先通过 Feature Flag 或动态配置灰度。
2. **失败可降级**：外部依赖失败时，尽量保留核心阅读链路。
3. **状态可解释**：Job、外部调用、用户可见失败都应有可追踪记录。
4. **最小暴露**：前端只暴露公开配置，敏感校验只在服务端执行。
5. **不以增长破坏体验**：任何监控和运营动作都受 `1-VISION.md` 的产品红线约束。

## ConfigKey 治理

动态参数统一走配置中心，不在业务代码中散落硬编码。

### 单一真相源

- 后端配置项在统一枚举 / 注册表中声明。
- 管理后台通过统一参数配置入口读写。
- 前端只消费后端显式下发的公开配置。

### 三处一致性

新增或修改配置项时，必须同时评估：

1. 后端配置声明和默认值；
2. 管理 API 的读取 / 保存 DTO；
3. 管理后台的表单、文案和权限展示。

如果配置影响用户可见体验，还需要同步产品文档或 UI 文案。

### 配置分组

配置按产品域或工程域分组，例如：

- 内容处理；
- 早报与推荐；
- AI 伴读与翻译；
- Pro 与订阅；
- 安全与限流；
- 可观测性；
- Mobile（当前暂缓，重启后再纳入重点运维面）；
- 外部集成。

禁止把临时实验配置长期留在通用分组中。

## Feature Flag

Feature Flag 用于灰度、回滚和能力边界控制。

### 使用准则

- 新能力默认关闭，经过测试、灰度和监控后再放量。
- 涉及付费、权限、安全、外部写操作的能力必须有关闭路径。
- Flag 关闭后，用户应看到稳定降级态，而不是 500 或空白页面。
- 删除过期 Flag 前需要确认所有环境和文档都已迁移。

### Kill Switch

高风险链路需要有全局关闭开关：

- AI 外部调用；
- 扣费 / 配额；
- 推送 / 邮件 / 外部发布；
- 大批量 Job；
- 新读模型或新索引；
- 影响 Pro 权益的能力。

Kill Switch 的语义必须清楚：关闭后是隐藏入口、只读、返回友好错误，还是停止后台任务。

## 可观测性

### 日志

- 日志用于定位问题，不用于存储业务数据。
- 禁止写入完整 token、密钥、连接字符串、完整邮箱、完整用户 ID 或长堆栈。
- 用户可见错误要有 requestId 或等价追踪标识。
- 外部调用失败要区分超时、限流、鉴权失败、上游错误和本地解析失败。

### 指标

关键路径至少覆盖：

- 请求成功率、错误率和延迟；
- 外部依赖调用次数、失败率和耗时；
- Job 执行结果、耗时、跳过原因和重试情况；
- Pro、配额、扣费、权限和限流相关事件；
- 核心产品漏斗事件。

指标命名应稳定、可聚合，避免把高基数字段放进标签。

### 事件

后端事件通过统一事件适配层上报，新增事件需要先登记到事件目录。事件 payload 只保留分析所需字段，不上传原文、隐私数据、完整异常或可反查用户身份的字段。

### 告警

告警统一通过告警分发服务触发，禁止业务代码直接调用具体通知渠道。

告警级别建议：

| 级别 | 语义 |
| --- | --- |
| CRITICAL | 用户已经明显受影响，或存在数据 / 权限 / 计费风险 |
| WARNING | 指标异常、失败率升高或即将影响用户 |
| INFO | 仅记录状态变化，不主动打扰 |

告警必须具备去重语义，避免同一问题在短时间内重复触达。

## 开发时的监控与告警要求

新增或修改以下内容时，必须同时评估监控和回滚：

### 新增 / 修改 Job

- Job 必须具备幂等性或清晰的重复执行保护。
- 多实例部署下只能由一个实例执行的任务必须加分布式锁。
- Job 需要记录开始、成功、失败、跳过和耗时。
- 长周期 Job 需要更宽松的健康判断，不能套用短周期任务阈值。

### 新增 / 修改外部调用

- 外部调用必须封装在 Adapter 边界内。
- 必须设置超时、重试上限和失败分类。
- 失败时应明确是降级、跳过、重试还是返回用户错误。
- 不允许把上游原始错误完整透传给用户或告警渠道。

### 新增 / 修改 API 端点

- 明确匿名、登录、Pro、管理员和内部调用的权限边界。
- 用户可控输入必须校验和限流。
- 敏感端点需要审计或操作记录。
- 返回格式保持统一 envelope。

### 新增事件 / 指标

- 命名需要稳定，避免一次性实验名污染长期指标。
- 高基数字段只进属性，不进指标标签。
- 涉及用户身份的字段必须脱敏或哈希。

### 新增告警

- 必须定义触发条件、恢复条件、去重维度和用户影响。
- 告警文案不包含敏感信息。
- 同类告警优先复用已有入口和分组。

## 灰度与回滚

推荐灰度顺序：

1. 本地和测试环境验证；
2. 内部账号验证；
3. 小流量灰度；
4. 扩大到目标用户；
5. 全量开放；
6. 删除临时开关或保留为长期 Kill Switch。

推荐回滚优先级：

1. 关闭 Feature Flag；
2. 调整动态配置；
3. 停止后台任务或外部写操作；
4. 数据修复或补偿；
5. 代码回滚。

每个大功能上线前都需要写清楚：关闭入口、数据影响、是否需要补偿、如何确认恢复。

## 部署与环境

### 环境原则

- 不同环境使用独立凭证和隔离数据边界。
- 生产凭证不进入本地、测试或公开文档。
- 新环境必须先完成索引、队列、缓存、对象存储和外部依赖的最小可用验证。
- 公开仓只保留变量用途说明，不写真实值、内网地址或账号命名。

### 必填配置

服务启动所需配置分为：

- 鉴权与加密；
- 数据库与缓存；
- AI / 内容处理；
- 邮件 / 通知；
- 支付 / 订阅；
- 监控与事件；
- 对象存储；
- Mobile 发布相关配置（当前暂缓，重启后再补齐）。

具体变量名以各组件 `.env.example` 和部署模板为准。

## 安全相关运维项

- Token 存储在 HttpOnly Cookie 或平台安全存储中，前端 JavaScript 不读取敏感 token。
- 前端请求经由受控代理层访问后端敏感 API。
- 内部端点必须有内部凭证或等价保护，不依赖路径隐藏。
- 用户可控重定向只允许同站相对路径。
- 新增代理路径、上传路径、抓取路径和外部 URL 输入必须做白名单或严格校验。
- 删除账号、订阅变更、支付回调和管理员操作需要审计。

## 外部依赖

外部依赖按影响分级：

| 等级 | 说明 | 处理原则 |
| --- | --- | --- |
| 强依赖 | 缺失会影响核心用户功能 | 明确降级或阻断策略，必须监控 |
| 中依赖 | 影响特定功能，核心阅读可继续 | 功能级关闭，提示用户稍后再试 |
| 弱依赖 | 可选增强或运营能力 | 静默降级或隐藏入口 |

依赖清单见 `13-INFRA-DEPS.md`，本文不维护供应商账号、真实拓扑和成本。

## 订阅源治理

订阅源治理需要兼顾质量、可用性和安全：

- 抓取失败不应影响其它来源；
- 长期不可用来源需要降级、提示或禁用；
- 用户私有来源不得进入公共质量池；
- URL 探测必须防 SSRF、重定向绕过和异常大响应；
- 来源质量变化需要可解释地反馈给用户或管理员。
